# Phase 4 図生成リファクタリング 実装計画

## 概要

Phase 4 の図生成処理を以下の方針で再設計する。

| 現状 | 変更後 |
|------|--------|
| Phase 3 成果物全体を LLM に渡す | 対象セクションのみをルールベースで抽出して LLM に渡す |
| LLM が図の生成 + 元文書の再出力を行う | LLM は PlantUML コードの生成のみを行う |
| LLM が挿入位置を判断して文書を組み立てる | ルールベースで図を対象セクション末尾に挿入する |

| ファイル | 変更種別 | 概要 |
|---------|---------|------|
| `src/summarizer/phases/phase4_diagram.py` | 大幅改修 | 処理フロー全体を再設計 |

## 新しい処理フロー

```
Phase3Result.document
    │
    ▼
【Step 1: セクション解析（ルールベース）】
  Markdown 見出しを解析し、対象セクションのテキストを切り出す
  ・### 2.1 内部構成
  ・### 2.2 外部依存関係
  ・## 4. データ定義（章全体）
  ・## 5. 処理フロー の冒頭テキスト（小機能ありの場合）または章全体（小機能なし）
  ・各 ### 5.X サブセクション（小機能ありの場合）
    │
    ▼
【Step 2: 図の生成（LLM）】
  各抽出セクションに対してLLMを呼び出す
  ・入力: セクションテキスト + 図種別プロンプト
  ・出力: ```plantuml ... ``` ブロックのみ（図不要な場合は空文字）
    │
    ▼
【Step 3: 図の挿入（ルールベース）】
  生成された PlantUML コードを対象セクション末尾に挿入する
  ・導入文（固定文言）+ PlantUML コードブロック
  ・空文字が返った場合は挿入しない
    │
    ▼
Phase4Result.document
```

## セクション抽出ルール

| 対象セクション | 抽出範囲 | 小機能有無による分岐 |
|--------------|---------|-------------------|
| `### 2.1 内部構成` | 該当 `###` 行から次の `###` または `##` の直前まで | なし |
| `### 2.2 外部依存関係` | 該当 `###` 行から次の `###` または `##` の直前まで | なし |
| `## 4. データ定義` | 該当 `##` 行から次の `##` の直前まで（章全体） | なし（小機能の有無に関わらず章全体を渡す） |
| `## 5. 処理フロー` 冒頭 | 該当 `##` 行から最初の `### 5.` の直前まで | 小機能あり |
| `## 5. 処理フロー` 全体 | 該当 `##` 行から次の `##` の直前まで | 小機能なし |
| `### 5.X <小機能名>` | 各 `### 5.X` 行から次の `###` または `##` の直前まで | 小機能あり |

**小機能の有無の判定**: 抽出した5章テキスト内に `### 5.` で始まる行が存在するかどうか

## 図種別プロンプトの方針

各プロンプトファイルのテンプレート変数は `{section_text}` のみ。
現状の `phase4_diagram.md` から以下の制約・ルールをすべて引き継ぐ。

- alias ルール（英小文字・アンダースコアのみ、`as alias` 必須など）
- 禁止記法（ショートハンド禁止、diagram 種別混在禁止など）
- 出力形式（` ```plantuml ``` ` ブロックのみ、不要な場合は空文字）
- 自己検証チェックリスト

| プロンプトファイル | 対象 | LLM への指示 |
|-----------------|------|------------|
| `phase4_class_diagram.md` | 2.1、4章 | クラス図を生成。情報不足の場合は空文字を返す |
| `phase4_component_diagram.md` | 2.2 | コンポーネント図を生成。情報不足の場合は空文字を返す |
| `phase4_activity_diagram.md` | 5章概要・全体 | アクティビティ図（全体フロー概要）を生成。情報不足の場合は空文字を返す |
| `phase4_subsection_diagram.md` | 各5.X | 記述内容に応じてアクティビティ図・状態遷移図・シーケンス図のいずれかを生成。情報不足または該当なしの場合は空文字を返す |

## 図の挿入ルール

- 生成された PlantUML コードが空でない場合のみ挿入する
- 挿入位置: 対象セクションの末尾（次の見出し行の直前の空行の後）
- 挿入内容: 導入文 1行（空行）+ PlantUML コードブロック（空行）

### 導入文（固定文言）

| 対象 | 導入文 |
|------|--------|
| 2.1 クラス図 | `以下に内部構成図を示す。` |
| 2.2 コンポーネント図 | `以下に外部依存関係図を示す。` |
| 4章 クラス図 | `以下にデータ構造図を示す。` |
| 5章概要 アクティビティ図 | `以下に全体処理フロー図を示す。` |
| 5.X アクティビティ図等 | `以下に処理フロー図を示す。` |

## 実装ステップ

### Step A: プロンプトファイルの新規作成（4ファイル）

現状の `phase4_diagram.md` から共通ルール（alias ルール・禁止記法・自己検証チェックリスト）を抽出し、
各図種別プロンプトに組み込む形で作成する。

1. `phase4_class_diagram.md` を作成する
2. `phase4_component_diagram.md` を作成する
3. `phase4_activity_diagram.md` を作成する
4. `phase4_subsection_diagram.md` を作成する

### Step B: `phase4_diagram.py` の改修

現状の `Phase4DiagramGenerator.run()` を以下の3メソッドに分割・再設計する。

1. **`_extract_diagram_targets(document: str) -> list[DiagramTarget]`**
   - Markdown を行単位で走査し、対象セクションのテキストと挿入位置（行番号）を収集する
   - `DiagramTarget` は対象セクションテキスト・プロンプトファイル名・導入文・挿入行番号を保持する

2. **`_generate_diagram(section_text: str, prompt_file: str) -> str`**
   - 指定プロンプトで LLM を呼び出し、PlantUML コードブロックを返す
   - 空文字が返った場合はそのまま空文字を返す

3. **`_insert_diagrams(document: str, targets: list[DiagramTarget], diagrams: list[str]) -> str`**
   - 挿入位置の後ろから順に（後ろから処理することで行番号のズレを防ぐ）導入文と PlantUML コードを挿入する

4. **`run()` の更新**
   - 上記3メソッドを順に呼び出す形に書き換える
   - トークン数・実行時間・呼び出し回数のメタデータ集計は維持する

### Step C: 動作確認

- Phase 3 成果物のサンプル（小機能あり・なし両パターン）で実行し、図が正しい位置に挿入されることを確認する
